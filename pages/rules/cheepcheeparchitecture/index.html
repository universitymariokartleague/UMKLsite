1. What this project is

Cheep Cheep is a single Python process that does two jobs at once:

  1. A discord.py bot (slash commands, buttons, modals) that runs in the UMKL Discord server, providing team management, match scheduling, results verification, leaderboards, live-stream graphics, captcha verification, etc.
  2. A Flask + Waitress HTTP API (services/dbserver.py), running in a background thread inside the same process, that serves match/team/season data to the umkl.co.uk website.

Both halves read and write the same SQLite database and JSON files, via the same core/logic.py module.

3. Layering rules (how imports actually flow)

  • main.py loads cogs, starts background services and owns discord.Client
    • cogs/* cogs that contain relevant features (see §6)
      • bot/* imports pagination.py → embed.py
      • services/* calls the service synchronously (such as backup.py, data_export.py, gitimport.py)
      • core/* logic.py (SQLite + JSON), graphics.py (Pillow), generate_calendar.py (matches.ics)
        • config.py imported by every module above, contains configuration data
    • services/* dbserver.py, checkrss.py, predictionpolls.py, gitimport.py (started directly by main.py as daemons, not via a cog)
      • dbserver.py the public Flask + Waitress HTTP API (api.umkl.co.uk) that powers this website — see §5.12
      • core/* via dbserver.py — same core/* as above
  • core/logic.py is the only module that touches SQLite or the JSON data files directly, nothing else writes to the database
  • services/dbserver.py keeps its own read-only connection to the database for the web API (see §5.11 and §6).

4. Startup sequence - main.py

  1. config.py loads .env via python-dotenv and exposes TOKEN, CHANNEL_IDS, file paths, etc...
  2. main.check_args() - if --gitclone is passed, calls services.gitimport.start_git_clone() to pull the latest code from git before boot (used for the self-update/restart flow, see §5.11 and §6)
  3. core.generate_calendar.build_calendar() - regenerates data/matches.ics once at boot from data/matchdata.json
  4. services.dbserver.start_db_server() - spawns two daemon threads: the Waitress HTTP server (port 5555, proxied externally as api.umkl.co.uk) and an avatar-refresh loop used in said API (see §5.12)
  5. A daemon thread runs services.checkrss.check_rss_loop, calling back into
    main.on_rss_update → posts new UMKL news links to the news-articles channel
  6. A daemon thread runs services.predictionpolls.polling_loop - this file is currently unfinished (see §5.10)
  7. bot.run(TOKEN) starts the Discord client.
    In PersistentViewBot.setup_hook():
    • Registers cogs.match_cog.SignupButton as a persistent dynamic item (so signup buttons keep working after a restart, without needing the original message's view state)
    • Loads every cog listed in main.py (order matters for dev_cog, see §6)
    • Registers cogs.listeners_cog.CaptchaButton as a persistent dynamic item
    • Injects update_signup_embed (from match_cog) and PredictionPollView (from listeners_cog) onto the bot object, then loads dev_cog last so it can read those two attributes in its constructor

5. Module reference

5.1 config.py

Loads .env and defines every constant used elsewhere:

  • Discord token/IDs/map of channel IDs
  • File paths for the DB and all JSON data files
  • Graphics constants (fonts, caching flags)
  • The full MK8DX track name list
  • Captcha/backup tuning values

Every other module imports from here directly; it imports nothing from the rest of the project.

5.2 core/logic.py (the data layer)

The point of contact with data/umkl_db.db (SQLite) and every JSON data file. Organised into sections:

  • Current season/match lookup
  • Registration
  • Points/standings (with both single-team and batched multi-team query variants - batched versions exist for speed and optimisation purposes)
  • Match lifecycle (create/edit/signup/substitute)
  • SP calculation
  • Livescore data
  • Prediction polls
  • Captcha sessions
  • Round pairings
  • Profile-unlock items
  • The personal data export query

Also defines TeamData (a namedtuple) used as the return type of get_team_data() throughout the codebase.

It only imports config and .generate_calendar.build_calendar (which is called after any matchdata.json write - see §5.4).

5.3 core/graphics.py

Pillow-based image generation for:

  • Team standings
  • Versus graphics
  • Results images
  • Winner images
  • Welcome images
  • Stream overlays
  • Room-code images
  • Captcha images
  • YouTube thumbnails
  • Round-schedule images

Maintains in-memory FONT_CACHE, IMAGE_CACHE, and URL_CACHE dicts (toggled via config.CACHING_ENABLED) so repeated renders don't reload fonts/backgrounds/team emblems from the disk every time. Calls into core.logic for all the data it renders (team names/colours, match results, standings). apply_team_gradient_overlay() recolours the stream-overlay background per-team by hue distance matching, used for /generate-stream-overlay.

The captcha text/image pipeline (generate_captcha_text(), create_captcha_image()) also lives here. cogs/listeners_cog.py initiates the captcha flow (challenging members on join).

5.4 core/generate_calendar.py

  • Reads data/matchdata.json and writes data/matches.ics (an RFC 5545 calendar, via the icalendar package, localised to Europe/London)
  • build_calendar() is called once at boot by main.py (from core.generate_calendar import build_calendar), and again by core.logic every time match data changes services/ (see §3)
  • services/dbserver.py imports its OUTPUT_PATH constant to serve the /matches.ics route

5.5 bot/embed.py

  • Builds every discord.Embed sent by the bot, grouped by feature (teams, profile, matches, live score, leaderboards, polls, rules, logs/passwords, captcha)
  • Reads whatever data it needs from core.logic
  • Leaderboard functions return (embed, total_pages) tuples, with total_pages worked out via logic.compute_total_pages(...)

5.6 bot/pagination.py

  • Two Views: Pagination (a paginator wrapping any get_page(index) async callable) and SeasonEmbedPagination (adds a season Select dropdown on top, used by every season-scoped leaderboard)
  • Used by cogs/user_profile_cog.py's /leaderboards command
  • Keeps a normal, module-level from . import embed to use embed.display_wrong_interaction_user()

5.7 cogs/

All ten cogs are loaded by main.py. See §6 for exactly which functions/classes each cog imports from another cog - cogs/match_cog.py in particular acts as a shared library of autocomplete providers and helpers for several other cogs.

5.8 services/backup.py

  • backup_database() copies the DB plus every JSON data file into a temp dir, then shells out to the external 7z binary to produce backups/YYYY/MM/DD/{timestamp}_backup.7z
  • Not a background loop - it's called synchronously, on demand
  • From listeners_cog.py's scheduled task (every config.BACKUP_INTERVAL minutes, within config.BACKUP_START_HOUR–config.BACKUP_END_HOUR)

5.9 services/checkrss.py

  • Polls https://umkl.co.uk/news.xml every 60s (using ETag caching to skip unchanged fetches)
  • Diffs new <item> links against data/seenrsslinks.json, and invokes a callback with any new items
  • Started as a daemon thread from main.py - the callback (main.on_rss_update) posts each new link to the news-articles channel

5.10 services/predictionpolls.py

  • A currently unfinished match prediction polling system

5.11 services/gitimport.py

  • Self-update mechanism: start_git_clone() clones config.GITHUB_LINK into a temp folder and recursively moves the files over the project root
  • restart_bot() runs that then calls sys.exit(1)
  • In production the bot runs on Linux, so that relauncher is systemd - the unit file runs with Restart=always, so when sys.exit(1) kills the process, systemd just starts it again, now running the freshly git-pulled code. tests/supervisor.py (see §8) only exists to stand in for systemd on Windows, where there's no (easy) service manager to do this

5.12 services/dbserver.py

  • The public web API - runs its own Flask app under Waitress on port 5555 (proxied externally as api.umkl.co.uk), started as a daemon thread by start_db_server() (called from main.py)
  • Uses its own get_db_connection() context manager (separate from core.logic.get_connection(), though pointed at the same data/umkl_db.db file) for a couple of routes
  • Delegates almost all actual data assembly to batched core.logic functions (get_teams_season_points, get_teams_wins_losses_season, get_teams_championships_data, etc.) so a single /teamdata request doesn't run one query per team
  • Also runs its own avatar-refresh daemon thread (_avatar_refresh_loop) that re-caches Discord avatars for the Credits page every 24 hours

The available routes:

Route Method Notes

/seasoninfo POST logic.get_season_status + logic.get_season_match_progress

A "match" is a tournament row, each competing team gets a tournament_entry row, and each signed-up player gets a tournament_result row:

  • Initially points = NULL and approved = 0
  • Filled in by /add-match-results and flipped to approved = 1 once a board member approves the results embed
  • tournament_result has no primary key of its own, so a player's row is only ever looked up by the tournament_entry_id/user_id pair
  • A team can incur at most one penalty row per entry: penalty.tournament_entry_id is both the primary key and the foreign key back to tournament_entry, so the relationship is strictly 1:1
  • Points shown are always "team total minus the team's penalty row for that entry" - see the various get_teams_*_points* functions in core/logic.py

8. Tests

A mix of unittests tests and manual testing scripts:

  • test_graphics.py - a real unittest.IsolatedAsyncioTestCase suite. Generates every core.graphics image type and pixel-diffs it (via Pillow + numpy, 0.99 similarity threshold) against a checked-in reference image in tests/output/
  • test_dbserver.py - a standalone benchmarking script that reimplements most of core.logic's team/season query functions locally, each wrapped in time.perf_counter() timing, to profile the SQL behind /teamdata
  • test_requests.py - manual smoke-test script that fires real HTTP requests at https://api.umkl.co.uk (every dbserver.py route) and prints status/timing/JSON

9. Worked example: /create-match end to end

Here's the call path for one command: cogs/match_cog.py's /create-match:

  1. match_cog validates the date/time and season membership using core.logic (get_current_season, get_teams_in_season, is_team_in_season).
  2. bot.embed.display_match_creation_progress_update renders a "Creating match..." progress embed (updated multiple times as steps complete).
  3. core.logic.create_match inserts the tournament row and create_new_match_entry is called once per team to insert tournament_entry rows.
  4. core.graphics.create_event_versus_image renders output/versus.png from the two teams' data (fetched again via core.logic.get_team_data).
  5. A Discord Scheduled Event is created directly via the discord.py API - its ID is stored back with core.logic.add_event_id_to_match.
  6. core.logic.add_to_matchdata appends the extended match info to data/matchdata.json, which internally calls core.generate_calendar.build_calendar() to keep matches.ics in sync.
  7. bot.embed.display_team_signup builds the sign-up embed (with the versus image attached), match_cog.SignupButton (a persistent discord.ui.DynamicItem) is attached as the view and posted to each team's announcements channel plus the schedule channel.
  8. services.backup.backup_database() is called before all of the above, since match creation mutates both the DB and JSON state.

Every other multi-step flow in the bot (results verification in results_cog.py, team creation in team_cog.py, season rollover in season_cog.py) follows the same shape:

  • Validate via core.logic
  • Mutate via core.logic
  • Render via core.graphics
  • Present via bot.embed
  • Back up via services.backup
