Cheep Cheep — Architecture & Code Guide

A technical, developer-facing guide to how the Cheep Cheep codebase is put together: what each module is responsible for, and how the code calls into itself

This page is aimed at maintainers of the @Cheep Cheep Discord bot and API codebase, not players. For a general, non-technical explanation of how Cheep Cheep works, see the Discord/Cheep Cheep FAQ; for the full slash-command list, see the Commands Documentation page. This document is about the code, not the product.

1. What this project is

Cheep Cheep is a single Python process that does two jobs at once:

  1. A discord.py bot (slash commands, buttons, modals) that runs the UMKL (University Mario Kart League) Discord server — team management, match scheduling, results verification, leaderboards, live-stream graphics, captcha verification, etc.
  2. A Flask + Waitress HTTP API (services/dbserver.py), running in a background thread inside the same process, that serves match/team/season data to the umkl.co.uk website.

Both halves read and write the same SQLite database and JSON files, via the same core/logic.py module — there is no separate backend service.

3. Layering rules (how imports actually flow)

                    ┌───────────┐
                    │  main.py  │  loads cogs, starts services, owns the discord.Client
                    └─────┬─────┘
                          │ imports
                          ▼
     ┌───────────────────────────────────────────┐
     │                  cogs/*                    │  one discord.py Cog per feature
     │  (also import EACH OTHER — see §6)          │
     └───────┬───────────────────────┬─────────────┘
             │ imports                │ imports
             ▼                        ▼
        ┌─────────┐             ┌───────────┐
        │  bot/*  │◄───────────►│  services/*│   (dbserver, backup, data_export, ...)
        │(embed,  │  imports      └─────┬─────┘
        │pagination)                    │ imports
        └────┬────┘                     ▼
             │ imports              ┌─────────┐
             └─────────────────────►│  core/*  │  logic.py (DB/JSON), graphics.py (Pillow)
                                     └────┬─────┘
                                          │ imports
                                          ▼
                                     config.py

core/logic.py is the only module that touches SQLite or the JSON data files directly. Every cog, every service, and bot/embed.py goes through it — nothing else opens data/umkl_db.db on its own (except cogs/sql_cog.py, the raw-SQL debug tool, and services/dbserver.py, which intentionally keeps its own read-only connection for the web API — see §5.11 and §6).

Note
One layering exception worth knowing about: core/logic.py imports services/generate_calendar.build_calendar and calls it after every mutation to matchdata.json (new match, edited match, renamed team, rescheduled match). So the "core" layer reaches down into "services" for that one thing — it's how the public matches.ics calendar feed stays in sync with the database.

4. Startup sequence (main.py)

  1. config.py loads .env via python-dotenv and exposes TOKEN, CHANNEL_IDS, file paths, etc.
  2. main.check_args() — if --gitclone is passed, calls services.gitimport.start_git_clone() to pull the latest code from git before boot (used for the self-update/restart flow, see §5.11 and §6).
  3. services.generate_calendar.build_calendar() — regenerates data/matches.ics once at boot from data/matchdata.json.
  4. services.dbserver.start_db_server() — spawns two daemon threads: the Waitress HTTP server (port 5555, proxied externally as api.umkl.co.uk) and an avatar-refresh loop.
  5. A daemon thread runs services.checkrss.check_rss_loop, calling back into main.on_rss_update → posts new UMKL news links to the news-articles channel.
  6. A daemon thread runs services.predictionpolls.polling_loop — currently a no-op stub (see §5.10).
  7. bot.run(TOKEN) starts the Discord client. In PersistentViewBot.setup_hook():
    • Registers cogs.match_cog.SignupButton as a persistent dynamic item (so signup buttons keep working after a restart, without needing the original message's view state).
    • Loads every cog listed in main.py (order matters for dev_cog, see §6).
    • Registers cogs.listeners_cog.CaptchaButton as a persistent dynamic item.
    • Injects update_signup_embed (from match_cog) and PredictionPollView (from listeners_cog) onto the bot object, then loads dev_cog last so it can read those two attributes in its constructor.

5. Module reference

5.1 config.py

Loads .env and defines every constant used elsewhere: Discord token/IDs/ channel map, file paths for the DB and all JSON data files, graphics constants (fonts, caching flags), the full MK8DX track name list, and captcha/backup tuning values. Every other module imports from here directly; it imports nothing from the rest of the project.

5.2 core/logic.py (2,786 lines — the data layer)

The single point of contact with data/umkl_db.db (SQLite) and every JSON data file. Organised into sections: current season/match lookup, registration, points/standings (with both single-team and batched multi-team query variants — the batched versions exist specifically so services/dbserver.py's /teamdata endpoint and core/graphics.py's standings image don't do one query per team), match lifecycle (create/edit/signup/substitute), SP (season-points-like currency) calculation, live score, polls, captcha sessions, round pairings, profile-unlock items, and the personal data export query. Also defines TeamData (a namedtuple) used as the return type of get_team_data() throughout the codebase.

Imports services.generate_calendar.build_calendar (called after any matchdata.json write) and config only — no cog or service imports.

5.3 core/graphics.py (1,612 lines)

All Pillow-based image generation: team standings, versus graphics, results images, winner images, welcome images, stream overlays, room-code images, captcha images, YouTube thumbnails, and round-schedule images. Maintains in-memory FONT_CACHE, IMAGE_CACHE, and URL_CACHE dicts (toggle via config.CACHING_ENABLED) so repeated renders don't reload fonts/backgrounds/ team emblems from disk every time. Calls into core.logic for all the data it renders (team names/colours, match results, standings). apply_team_gradient_overlay() recolours the stream-overlay background per-team by hue distance matching, used for /generate-stream-overlay.

5.4 core/captcha.py

17-line wrapper: generate_captcha_text() picks random characters, and generate_captcha_image(text) just delegates to core.graphics.create_captcha_image. Used exclusively by cogs/listeners_cog.py.

5.5 bot/embed.py (1,580 lines)

Builds every discord.Embed sent by the bot, grouped by feature (teams, profile, matches, live score, leaderboards, polls, rules, logs/passwords, captcha). Reads whatever data it needs straight from core.logic (it is not passed pre-fetched data by the cogs — each display_* function does its own lookups). Leaderboard functions return (embed, total_pages) tuples designed to be fed straight into bot.pagination.Pagination. Imports Pagination from .pagination for display_wrong_interaction_user()'s file attachment — meaning bot/embed.py and bot/pagination.py import from each other (embed → pagination for the class, pagination → embed for that one error embed).

5.6 bot/pagination.py

Two Views: Pagination (generic Back/Next paginator wrapping any get_page(index) async callable) and SeasonEmbedPagination (adds a season Select dropdown on top, used by every season-scoped leaderboard). Used by cogs/user_profile_cog.py's /leaderboards command.

5.7 cogs/ — one discord.py extension per feature

All ten cogs are loaded by main.py. See §6 for exactly which functions/ classes each cog imports from another cog — cogs/match_cog.py in particular acts as a shared library of autocomplete providers and helpers for several other cogs, despite being structured as an ordinary Cog itself.

Cog Responsibility
core_cog.py Bot lifecycle (/restart → services.gitimport.restart_bot()), tree sync, gated board/media password commands
dev_cog.py Legacy !-prefix admin/dev tools for exercising core.graphics/core.logic directly, plus two slash commands
listeners_cog.py Member join/leave (captcha flow), per-message SP awards, disconnect-keyword detection, scheduled DB backups (@tasks.loop)
live_score_cog.py Admin commands to add/edit/clear the live per-race score graphic data
match_cog.py Match creation/editing, the signup flow (button + modal), room codes, substitutions — the largest cog, and a dependency of five others
results_cog.py Score submission modal, verification/approve/reject/penalty workflow
season_cog.py /start-season, /start-next-round, /add-next-round-pairings
sql_cog.py Admin-only raw SQL execution against the DB — bypasses core.logic entirely
team_cog.py Team creation (Discord role/channels/category), colour/name changes, season entry, stream keys
user_profile_cog.py /user-profile, /team-profile, /leaderboards, data export, stream overlay & YouTube thumbnail generation

5.8 services/backup.py

backup_database() copies the DB plus every JSON data file into a temp dir, then shells out to the external 7z binary to produce backups/YYYY/MM/DD/{timestamp}_backup.7z. Not a background loop — it's called synchronously, on demand, from listeners_cog.py's scheduled task (every config.BACKUP_INTERVAL minutes, within config.BACKUP_START_HOUR–BACKUP_END_HOUR) and from several cogs right before any operation that mutates match/team/season state (match_cog, results_cog, season_cog, team_cog).

5.9 services/checkrss.py

Polls https://umkl.co.uk/news.xml every 60s (using ETag caching to skip unchanged fetches), diffs new <item> links against data/seenrsslinks.json, and invokes a callback with any new items. Started as a daemon thread from main.py; the callback (main.on_rss_update) posts each new link to the news-articles channel.

5.10 services/predictionpolls.py

A stub. polling_loop() is just pass — it returns immediately, so the daemon thread main.py starts for it exits right away and currently does nothing. The actual poll posting/voting UI (PredictionPollView) lives in cogs/listeners_cog.py and is triggered manually today (via dev_cog.py's test_poll), not by this service.

5.11 services/gitimport.py

Self-update mechanism: start_git_clone() clones config.GITHUB_LINK into a temp folder and recursively moves the files over the project root; restart_bot() runs that then calls sys.exit(1). This only does something useful in combination with tests/supervisor.py (see §8), which relaunches main.py whenever it exits — that's what /restart (in core_cog.py) actually relies on to pick up new code.

5.12 services/generate_calendar.py

Reads data/matchdata.json and writes data/matches.ics (an RFC 5545 calendar, via the icalendar package, localised to Europe/London). build_calendar() is called once at boot by main.py and again by core.logic every time match data changes — see the layering note in §3.

5.13 services/dbserver.py

The public web API. Runs its own Flask app under Waitress on port 5555 (proxied externally as api.umkl.co.uk), started as a daemon thread by start_db_server() (called once from main.py). Uses its own get_db_connection() context manager (separate from core.logic.get_connection(), though pointed at the same data/umkl_db.db file) for a couple of routes, but delegates almost all actual data assembly to batched core.logic functions (get_teams_season_points, get_teams_wins_losses_season, get_teams_championships_data, etc.) so a single /teamdata request doesn't run one query per team. Also runs its own avatar-refresh daemon thread (_avatar_refresh_loop) that re-caches Discord avatars for a hardcoded credits list every 24h.

Routes (full request/response shapes are documented in the bot repo's README.md):

Route Method Notes
/teamdata POST Delegates to ~13 batched core.logic lookups, one per team field
/seasoninfo POST logic.get_season_status + logic.get_season_match_progress
/teamcolors POST Direct query, enabled teams only
/teamlocations POST logic.get_teams_coords, optionally filtered by season
/matchdata POST logic.get_matchdata() enriched with live-computed results/penalties for pending matches
/matches.ics GET Serves the file written by services.generate_calendar.build_calendar()
/live POST logic.get_live_score()
/pfps/<name> GET Serves cached credits-team avatars
/teamemblems/<name> GET Serves team emblem assets (falls back to DEFAULT)

5.14 services/data_export.py

create_user_data_export(user_id, dest_dir) calls core.logic.get_user_data_export (match results, SP, poll votes) and zips it as user_data.json. Called from cogs/user_profile_cog.py's /export-my-data command.

6. Cross-cog import map

Most cogs only import config, bot.embed, core.logic, core.graphics, and services.* — a clean one-way dependency into the shared layers. The exceptions, where cogs import from each other, are:

Importer Imports from cogs.match_cog Why
main.py SignupButton, update_signup_embed Registers the button as a global persistent dynamic item; injects the embed-refresh function onto the bot for dev_cog
cogs/live_score_cog.py live_score_autocomplete, live_score_edit_autocomplete, track_name_autocomplete Parameter autocomplete for its three live-score commands
cogs/results_cog.py get_match_lock, match_conditional_team_autocomplete, open_match_autocomplete, team_open_match_autocomplete, update_signup_embed get_match_lock guards the Approve button against double-approval races; the rest are autocompletes; update_signup_embed refreshes the roster embed after results affect a match
cogs/team_cog.py enabled_teams_autocomplete Team-picker autocomplete on /change-team-colour
cogs/user_profile_cog.py during_matches_autocomplete, enabled_teams_autocomplete Autocomplete for /generate-yt-thumbnails and /team-profile

cogs/dev_cog.py doesn't import from match_cog/listeners_cog directly — instead main.py passes update_signup_embed and PredictionPollView into its constructor, which is why dev_cog must be load_extension'd last in main.py's setup_hook (both symbols need to already exist on the bot object).

This makes cogs/match_cog.py the de facto shared library for match-related autocompletion and locking, despite being structured as an ordinary Cog like the rest.

7. Data storage

SQLite (data/umkl_db.db)

season(season_id)
season_entry(season_entry_id, season_id → season, team_id → team)
team(team_id, team_name, team_full_name, team_color, team_markdown, place_id → uk_places, enabled)
uk_places(place_id, place_name, place_ascii, latitude, longitude, country, admin_name)
game(game_id, game_name, console_name, console_code, game_year)
tournament(tournament_id, season_id → season, tournament_date, game_id → game,
           start_time, event_id, test_match)
tournament_entry(tournament_entry_id, tournament_id → tournament, team_id → team,
                  signup_message_id)
tournament_result(tournament_entry_id → tournament_entry, user_id → user, points, approved)
penalty(tournament_entry_id → tournament_entry [PK], penalty_amount)
user(user_id)

A "match" is a tournament row; each competing team gets a tournament_entry; each signed-up player gets a tournament_result row (initially points = NULL, approved = 0) that's filled in by /add-match-results and flipped to approved = 1 once a board member approves the results embed. Points shown anywhere are always "team total minus that team's penalty row for that entry" — see the various get_teams_*_points* functions in core/logic.py.

JSON files (data/)

File Config constant Written by Purpose
matchdata.json MATCHDATA_PATH core.logic (add_to_matchdata, update_matchdata, reschedule_match, update_team_name) Extended match info not in SQLite: title, description, YouTube links, Discord event ID/URL, live-computed results snapshot
playersp.json PLAYERSP core.logic.update_player_sp Per-player SP totals (matches_sp + chat_sp) and history
liveresults.json LIVE_SCORE core.logic (add_to_live_score, edit_live_score, clear_live_score) Per-race 1st–12th place data for the live stream overlay/API
rounddata.json ROUND_DATA core.logic.save_round_data (via season_cog.py) Current round number + team pairings
polldata.json POLLDATA core.logic.save_poll_data Prediction poll votes
captchadata.json CAPTCHA_DATA core.logic (set_captcha_session, clear_captcha_session) In-progress captcha challenges per user
allowedgameids.json ALLOWED_GAME_IDS core.logic.save_allowed_game_ids Which game_ids /create-match may use
profileunlockitems.json PROFILE_UNLOCK_ITEMS (read-only from code) Cosmetic unlock flags encoded into the profile-site URL

streamkeys.json (read directly by team_cog.py) manual Per-team livestream keys for /request-stream-key

8. tests/

Not a uniform pytest suite — a mix of real tests and manual scripts:

  • test_graphics.py — a real unittest.IsolatedAsyncioTestCase suite. Generates every core.graphics image type and pixel-diffs it (via Pillow + numpy, 0.99 similarity threshold) against a checked-in reference image in tests/output/.
  • test_dbserver.py (830 lines) — not pytest-discoverable; a standalone benchmarking script that reimplements most of core.logic's team/season query functions locally, each wrapped in time.perf_counter() timing, to profile the SQL behind /teamdata.
  • test_requests.py — manual smoke-test script that fires real HTTP requests at https://api.umkl.co.uk (every dbserver.py route) and prints status/timing/JSON.
  • test.py — a scratch script calling a couple of core.logic functions directly.
  • supervisor.py — not a test at all: a process supervisor that loops forever, running main.py as a subprocess and immediately relaunching it on exit (a systemd-restart-on-exit stand-in for Windows). This is what makes /restart (core_cog.py → services.gitimport.restart_bot() → sys.exit(1)) actually useful — the exit is caught and the freshly- git-pulled code gets relaunched.

9. Worked example: /create-match end to end

To make the module interactions concrete, here's the call path for one command, cogs/match_cog.py's /create-match:

  1. match_cog validates the date/time and season membership using core.logic (get_current_season, get_teams_in_season, is_team_in_season).
  2. bot.embed.display_match_creation_progress_update renders a "Creating match..." progress embed (shown multiple times as steps complete).
  3. core.logic.create_match inserts the tournament row; create_new_match_entry is called once per team to insert tournament_entry rows.
  4. core.graphics.create_event_versus_image renders output/versus.png from the two teams' data (fetched again via core.logic.get_team_data).
  5. A Discord Scheduled Event is created directly via the discord.py API; its ID is stored back with core.logic.add_event_id_to_match.
  6. core.logic.add_to_matchdata appends the extended match info to data/matchdata.json, which internally calls services.generate_calendar.build_calendar() to keep matches.ics in sync.
  7. bot.embed.display_team_signup builds the sign-up embed (versus image attached); match_cog.SignupButton (a persistent discord.ui.DynamicItem) is attached as the view and posted to each team's announcements channel plus the schedule channel.
  8. services.backup.backup_database() is called before all of the above, since match creation mutates both the DB and JSON state.

Every other multi-step flow in the bot (results verification in results_cog.py, team creation in team_cog.py, season rollover in season_cog.py) follows the same shape: validate via core.logic → mutate via core.logic → render via core.graphics → present via bot.embed → back up via services.backup.
